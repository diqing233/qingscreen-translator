# 结果条停止/清空按钮设计
日期：2026-03-11

## 背景

当前结果条顶部工具栏缺少一个快速中断当前翻译流程、或在空闲时直接清空当前结果内容的入口。

用户确认的目标行为：

1. 在三角框选按钮右侧新增一个小方块图标按钮。
2. 按钮不显示长文字，只通过 tooltip 解释作用。
3. 按钮是双态：
   - 有翻译任务进行中时：执行“终止翻译”
   - 没有翻译任务时：执行“清空当前翻译内容”
4. 默认结果条宽度要拉长，尽量让顶部按钮在默认宽度下完整显示，不依赖横向滚动。

## 设计决策

### 按钮位置与样式

新增按钮放在顶部工具栏最左侧操作区，紧跟在三角框选按钮右边，位于“恢复默认大小”按钮左边。

按钮使用小方块图标 `■`，保持小尺寸图标按钮风格，不使用文字标签。

视觉状态：

- 空闲时：中性灰色，tooltip 为“清空当前翻译内容”
- 翻译中：警示色，tooltip 为“终止当前翻译”

### 行为模型

结果条只负责发出“停止/清空请求”信号，不直接控制后台线程。

控制器根据当前状态分流：

- 若存在活动中的翻译 worker：
  - 立即将对应任务标记为已取消
  - 请求 worker 中断
  - 立即清空结果条当前内容并把按钮恢复为空闲态
  - 若后台请求稍后返回，结果会被丢弃，不再刷新 UI
- 若当前没有活动中的翻译 worker：
  - 仅清空结果条当前翻译内容

### 取消策略

当前 `TranslationWorker` 内部调用的是同步翻译路由，没有现成的强制终止网络请求机制。

因此这次采用“请求中断 + 丢弃结果”的安全实现：

1. 控制器给活动 worker 发送 `requestInterruption()`
2. worker 在发信号前检查是否已被中断
3. 控制器额外维护“已取消任务 ID 集合”
4. 即使后台调用已经返回，只要任务被标记取消，就不会再更新 UI 或写入历史

这样能做到：

- 用户点击后界面立即结束等待
- 不使用 `terminate()` 这类危险线程终止方式
- 不会把“已经取消”的结果又刷回结果条

### 清空内容范围

“清空当前翻译内容”只清空结果条当前显示内容，不关闭翻译框，不清理历史，不影响托盘或设置状态。

清空后结果条恢复为默认占位状态：

- 译文区显示“等待翻译...”
- 原文区收起
- 来源标签清空
- 当前结果对象置空
- AI 科普区隐藏

### 默认宽度

当前默认宽度不足以在固定/多框模式下完整显示顶部工具栏。

本次直接增大 `DEFAULT_W`，目标是默认宽度下：

- 左侧滚动工具栏内容在常见状态下完整显示
- 右侧固定图标按钮仍有足够空间

保持横向滚动能力不变，仅减少默认触发滚动的概率。

## 代码边界

### `src/ui/result_bar.py`

变更内容：

- 新增停止/清空按钮和对应信号
- 新增双态按钮状态更新方法
- 新增清空当前结果内容方法
- 调整默认宽度常量
- 把新按钮插入到框选按钮右侧
- 工具栏尺寸刷新逻辑覆盖新按钮

### `src/core/controller.py`

变更内容：

- 跟踪活动中的翻译 worker
- 为翻译任务分配 request id
- 响应结果条停止/清空请求
- 取消后忽略结果与错误回调
- 翻译 worker 完成时同步刷新结果条按钮状态

### `src/ocr/ocr_worker.py`

变更内容：

- `TranslationWorker` 在发出结果或错误前检查线程中断状态

## 测试要点

1. 新按钮位于三角框选按钮右侧。
2. 默认宽度下，固定模式工具栏内容不再比滚动区视口更宽。
3. 控制器在有活动翻译任务时点击按钮，会请求中断并清空内容。
4. 控制器在空闲时点击按钮，只清空内容，不触发中断。
5. 已取消的翻译结果不会再刷新结果条或写入历史。
